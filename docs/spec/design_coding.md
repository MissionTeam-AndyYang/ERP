# 開發準則
此規範適用於前端畫面開發與後端 API 實作。

## 1. 開發實作要求
- 在開始實作前，完整分析專案架構，並理解此功能在整個系統中的定位  
- 不僅針對當前需求開發，還要考量未來相似流程與功能的擴充性、可維護性、模組重用  
- 在需求不足以完成抽象設計時，先提出可延伸的架構方案，再依照該架構完成本次功能  
- 在完成後檢查是否存在可共用卻重複的程式，若有則立即進行重構  
- 在程式設計與重構過程中同時考量效能最佳化，避免不必要的資源消耗或效能瓶頸，並確保程式碼在維護性與可讀性之間取得平衡  
- 執行相關測試並修正所有問題，確保功能正確性與穩定性  
- 在後續針對已制定規格之 **Database Schema 文件** 與 **API 文件** 進行修正時，同步更新已實作之相關後端 API 與前端串接程式，確保系統一致性  
---

## 2. Coding 原則
- **優先順序**：Correctness → Maintainability → Readability → Consistency → Performance  
- 不得為了縮短程式碼而降低可維護性  
- 不得為了效能而犧牲程式可讀性  

---

## 3. Refactoring 準則
完成需求後，必須針對所有已實作程式碼進行檢查與重構：
- 檢查是否存在重複程式，並將其統一包裝為 util_xxx() 函式或抽取為物件。  
- 檢查是否可抽象化  
- 檢查是否可共用  
- 檢查是否可改善命名  
- 檢查是否可提升可讀性  

> 若改善不影響功能，應立即進行重構。

---

## 4. 開發工作流程

### 後端API 開發流程規範
此流程適用於 API 設計、實作與前端串接，需遵循以下步驟：

**1：後端API 設計與提案**  
- 由 後端 負責 API 設計，並產出以下文件：  
  - API 提案文件  
  - 後端流程與演算法文件（供工程師 review 後再進行實作）  
  - 前端靜態預覽頁（供工程師評估 API 設計是否符合實際使用情境）  
- 以上文件須統一集中放置於 `docs\spec\api-proposal\` 目錄下。  

**2： 後端API 設計準則**
- 在設計後端 API 時，必須遵循 **前後端分離架構** 的經典原則：**Separation of Concerns（職責劃分）**。  
- 後端 API 的主要責任是提供正確、完整的資料與業務邏輯，不應承擔前端顯示或格式化的工作。  
- **必須思考哪些處理適合在前端完成**，並將這些責任留給前端，而不是全部交由後端處理。  
- 例如：  
  - **Enum → 字串轉換**：後端僅需回傳 Enum 值，字串轉換與多國語言支援應由前端處理。  
  - **前端導向的 query string**：前端應負責保存參數數值，並在需要時組合成導向使用者操作的 Query String，以確保邏輯與顯示層責任劃分清晰。  
- 此原則可避免後端過度耦合 UI 邏輯，提升系統的擴充性、可維護性與重用性。  

**3：後端API 實作與文件更新**  
- 僅在工程師確認 API 提案後，方可進行 後端API 實作與 API 文件更新。  
- 若尚未接收到工程師確認，但收到「進行實作」的需求，需先再次確認工程師是否要進行 API 實作。  
- 程式碼實作必須符合 **API 提案文件** 與 **後端流程與演算法文件** 的設計邏輯。  

**4：前端與 後端API 串接**  
- 僅在工程師確認 API 程式碼後，方可進行前端與 API 串接。  
- 前端在進行 Enum 字串轉換時，同時支援多國語言字串的轉換。
- 進行 planned_screen_list_naming.md 文件更新

---

### 後端API 實作檢查規範
1. **欄位邏輯驗證**  
   - 程式碼完成後，需檢查所有回傳欄位是否有正確計算或自資料庫取值回填。  
   - 若欄位僅定義但未處理，視為程式邏輯錯誤。  

2. **自動化測試**  
   - 為每個 API 撰寫完整的 **pytest 測試案例**，涵蓋：  
     - 欄位存在性檢查  
     - 欄位值正確性（計算／資料庫取值）  
     - 異常情境（空值、錯誤輸入）  
   - 測試需全部通過，否則不得提交。  
   - 產生測試報告文件，並統一放置於 `docs\report\` 目錄下，作為驗證紀錄。  

3. **自我驗證流程**  
   - Codex 完成程式碼後，必須自行執行測試並修正所有失敗案例。  
   - 測試結果需再次驗證，直到所有案例通過。  

4. **交付前檢查清單**  
   - 確認 API 文件與程式碼一致。  
   - 確認所有欄位邏輯已實作並測試。  
   - 確認程式碼可正常運作且無錯誤。  
   - 確認測試報告已生成並正確存放於 `docs\report\` 目錄下。  

## 5. 程式碼與資料庫風格規範
- 必須使用一致的命名規則

## 6. 開發演進準則
在開發過程中不僅需完成需求，還必須持續演進與改善，包含以下面向：
- **必須**提出設計建議 (Design Suggestions)，協助架構與流程持續優化  
- **必須**補齊並更新相關文件，保持文件與程式一致  
- **必須**主動發現並修正程式中的 Bug  
- **必須**補齊並維護測試案例，確保功能正確性
